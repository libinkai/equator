# 杂谈

## 思想的演进

> oriented 面向

1. POP 面向过程编程，线性思维
2. OOP 面向对象编程，封装、继承、多态；对象交互以完成功能（OOP的静态问题：类是一个固定的实体，可以修改属性值，覆写行为，替换抽象实现，但是类本身是稳定不变的即不便于拓展）
3. AOP 面向切面编程，在不修改类的情况下拓展功能，是OOP的补充，缓解OOP的静态问题
4. DDD 领域驱动设计，解决系统规模日益庞大导致OOP无法胜任的问题，将系统划分为各个盒子即领域。（POP没有边界、OOP有边界）

## DDD 项目结构

1. User Interface 展示
2. Application Layer 请求转发，处理跨领域调度
3. Domain Layer 业务+数据
4. Infrastructure Layer 基础设施

# 微服务为什么要选择DDD

## 软件架构模式的演进

- 单机架构：面向过程，UI层+数据层，CS架构
- 集中式架构：面向对象，业务接入层+业务逻辑层+数据库层，三层架构MVC或者SOA架构
- 分布式微服务架构：实现应用之间的解耦，解决单体应用拓展性和弹性伸缩能力不足的问题

## 微服务设计和拆分的困境

- 微服务拆分困境的根本原因就是不知道业务或者微服务的边界在什么地方
- 领域驱动设计，Domain Driven Design，DDD。DDD的核心思想是通过领域驱动设计方法定义领域模型从而确定业务和应用边界，保证业务模型与代码模型的一致性

## DDD为什么适合微服务

- DDD不是架构而是一种架构设计方法论
- DDD包括战略设计和战术设计两部分
